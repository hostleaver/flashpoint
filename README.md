# Flashpoint quest contributions

This repository accepts **working DayZ Expansion quest submissions** for Flashpoint.
It is not an idea board. A pull request should contain the quest JSON, every new
objective it references, and enough context for another person to review and test it.

## Before you start

- Build for the Flashpoint Livonia server.
- Use DayZ Expansion quest configuration, not prose alone.
- Keep one quest or one closely related quest chain per pull request.
- Use real DayZ class names and Livonia coordinates. State every required mod.
- Do not edit another contributor's submission.
- Do not submit generated player/group data or `PersistentServerData.json`.

Read the upstream documentation before authoring a quest:

- [Expansion Quests overview](https://github.com/salutesh/DayZ-Expansion-Scripts/wiki/%5BServer-Hosting%5D-Expansion-Quests)
- [Quest configuration](https://github.com/salutesh/DayZ-Expansion-Scripts/wiki/%5BServer-Hosting%5D-Quest-Configuration)
- [Objective configuration](https://github.com/salutesh/DayZ-Expansion-Scripts/wiki/%5BServer-Hosting%5D-Quest-Objectives-Configuration)
- [Quest NPC configuration](https://github.com/salutesh/DayZ-Expansion-Scripts/wiki/%5BServer-Hosting%5D-Quest-NPC-Configuration)

## Submit a quest

1. Fork this repository and create a branch.
2. Copy one of the packages in [`examples/`](examples/) to
   `submissions/<quest-slug>/`.
3. Replace its proposal and JSON with your implementation.
4. Validate all JSON and search the repository for ID collisions.
5. Open a pull request and complete the checklist.

Use lowercase kebab-case for `<quest-slug>`. Example:

```text
submissions/medical-supply-run/
```

## Package layout

The package root mirrors the useful parts of the server profile directory
`ExpansionMod/Quests`. Only include directories your quest uses.

```text
submissions/<quest-slug>/
|-- PROPOSAL.md
|-- Quests/
|   `-- <id>_<name>.json
|-- Objectives/
|   |-- Action/
|   |-- AICamp/
|   |-- AIPatrol/
|   |-- AIVIP/
|   |-- Collection/
|   |-- Crafting/
|   |-- Delivery/
|   |-- Target/
|   |-- Travel/
|   `-- TreasureHunt/
|-- NPCs/                              # only for a new quest NPC/object
`-- Mission/                           # only for mission-side assets
    `-- expansion/quests/objects/      # quest object-set .map files
```

Do not add empty objective directories. `ObjectiveType` must match its directory:

| Directory | Type | Purpose |
| --- | ---: | --- |
| `Target` | 2 | Kill configured targets |
| `Travel` | 3 | Reach or leave a location |
| `Collection` | 4 | Collect items |
| `Delivery` | 5 | Deliver items |
| `TreasureHunt` | 6 | Find a spawned treasure |
| `AIPatrol` | 7 | Clear an AI patrol |
| `AICamp` | 8 | Clear an AI camp |
| `AIVIP` | 9 | Escort or protect an AI NPC |
| `Action` | 10 | Execute configured actions |
| `Crafting` | 11 | Craft configured items |

## IDs and references

- Quest IDs are globally unique.
- Objective IDs are unique within an objective type. A Travel objective and a
  Target objective may share an ID; two Travel objectives may not.
- NPC IDs are globally unique.
- Before choosing an ID, search all JSON in this repository for the exact number.
- Use a provisional ID from `10000` through `99999`. Maintainers may renumber it
  when integrating the quest with the live server configuration.
- A quest's `Objectives` entry must repeat the referenced objective's `ID`,
  `ObjectiveType`, and current `ConfigVersion` exactly.
- `QuestGiverIDs`, `QuestTurnInIDs`, `PreQuestIDs`, and `FollowUpQuest` must resolve
  to files in the package or to an existing Flashpoint ID named in `PROPOSAL.md`.

Do not change `ConfigVersion` to make an old file look current. Start from a current
example or a file generated by the server.

## What the proposal must explain

Complete [`templates/PROPOSAL.md`](templates/PROPOSAL.md). Reviewers need:

- contributor names for credit beyond the Git history;
- the exact player flow from accepting through completion;
- quest giver, turn-in, prerequisites, cooldown, and group behavior;
- reward and balance reasoning;
- all coordinates and how they were checked in Livonia;
- mod dependencies and every non-vanilla class name;
- test steps and the result of an actual server test, if one was performed.

Screenshots are useful for locations, object placement, and NPC presentation, but
they do not replace the configuration files.

## Validate before opening a PR

Run this from the repository root in PowerShell:

```powershell
$failed = $false
$files = Get-ChildItem -Recurse -Filter *.json
foreach ($file in $files) {
    try {
        Get-Content -Raw -LiteralPath $file.FullName | ConvertFrom-Json | Out-Null
        "OK  $($file.FullName)"
    }
    catch {
        $failed = $true
        Write-Error "INVALID JSON: $($file.FullName): $_"
    }
}
if ($failed) { exit 1 }
```

Then review your diff and run `rg -n '"ID": 12345'` for each proposed quest,
objective, and NPC ID. JSON being syntactically valid does not prove that class
names, references, or coordinates work.

## Examples and source basis

- [`examples/collect-apples`](examples/collect-apples/) demonstrates a Collection
  objective and a normal quest.
- [`examples/craft-fishing-rod`](examples/craft-fishing-rod/) demonstrates a
  Crafting objective and an item reward.

Their schema, defaults, folder names, and objective type values were checked against
DayZ Expansion 1.9.68, including:

- `DayZExpansion_Quest_Constants.c`
- `ExpansionQuestConfig.c`
- `ExpansionQuestObjectiveType.c`
- `ExpansionDefaultQuestData.c`
- `ExpansionDefaultObjectiveData.c`

The examples use reserved IDs `90001` and `90002`; do not reuse those IDs in a
submission. Example dialogue and proposal text are original to this repository.
